Key Technical Points

01

동시 예약 중복 방지 — SELECT FOR UPDATE 비관적 락

동일 객실·날짜에 다수 요청이 동시 도달할 경우 재고가 음수로 차감되는 Race Condition 문제를 식별했습니다.

MyBatis에서 SELECT ... FOR UPDATE로 해당 row에 배타 락을 걸어 트랜잭션 완료 전까지 다른 트랜잭션의 접근을 차단했습니다. 재고 0 확인 시 RuntimeException을 발생시켜 @Transactional 전체 롤백으로 원자성을 보장했습니다.

낙관적 락 대신 비관적 락을 선택한 이유는 예약 도메인 특성상 충돌 빈도가 높아 롤백 재시도 비용보다 선점 대기가 유리하다고 판단했기 때문입니다.

02

회원 유형별 알림 분기 + Spring Scheduler 리마인드

예약 완료 후 회원 유형에 따라 알림 채널을 분기했습니다. 일반 회원은 JavaMail(SMTP) 이메일, 비회원은 CoolSMS API 문자를 발송하며, 알림 로직을 별도 서비스 계층으로 분리했습니다.

@Scheduled cron 표현식으로 매일 자정 체크인 전일 예약 건을 조회해 리마인드 메일을 자동 발송하는 배치성 스케줄러를 구현했습니다.

03

비즈니스 로직 중심 DB 테이블 설계 — 역할별 분리 구조

단순 예약 저장을 넘어 예약 상태 관리, 재고 차감, 시즌별 요금 정책, 프로모션 적용 등 복잡한 비즈니스 로직을 수용하기 위해 역할 단위로 테이블을 분리했습니다.

객실 정보 / 재고 분리
ROOM_TYPE ROOM_STOCK

객실 기본 정보(타입·사이즈·정원)는 변경 빈도가 낮아 ROOM_TYPE에 고정.
날짜별 가용 수량은 ROOM_STOCK으로 분리해 재고 변동을 독립적으로 관리.

시즌 / 요일 요금 정책
SEASON WEEKDAY_RATE

성수기·비수기 기간은 SEASON, 요일별 가격 정책은 WEEKDAY_RATE로 분리.
같은 객실이어도 날짜·요일 조합에 따라 요금이 동적으로 계산되도록 설계.

프로모션 다중 적용
PROMOTION_MASTER RESERVATION_PROMOTION_MAPPING

프로모션 정책은 PROMOTION_MASTER에 별도 관리.
매핑 테이블로 하나의 예약에 여러 할인 정책을 연결할 수 있는 N:M 구조로 설계.

예약 중심 데이터 연결
RESERVATION ROOM_STOCK PAYMENT

RESERVATION을 중심으로 재고·결제·프로모션 데이터를 연결.
예약 상태(대기·확정·취소)와 결제 정보를 분리해 환불 처리 시 독립적 롤백 가능.

⚠ 예약 가능 여부 판단 — 가장 복잡했던 부분

  • 단순 객실 존재 여부가 아닌 ROOM_STOCK에서 해당 날짜 재고 잔여 여부를 먼저 확인해야 했음
  • 같은 객실이어도 SEASON + WEEKDAY_RATE 데이터를 함께 조회해야 정확한 금액 산출 가능
  • 적용 가능한 프로모션 존재 여부까지 동시에 확인해야 최종 예약 금액 확정 가능
  • 결과적으로 단일 테이블 기준 처리 불가 → RESERVATION 중심으로 재고·시즌·프로모션을 JOIN하는 구조로 설계

04

K6 부하테스트 — XSSF vs SXSSF 성능 비교

대용량 엑셀 다운로드 API(10,112건)에 대해 K6로 동시 사용자 10명 → 50명 시나리오로 부하테스트를 진행했습니다. XSSF(기존)는 평균 응답시간 89ms로 빠르지만 전체 데이터를 힙에 올리는 구조상 동시 50명 + 힙 64MB 제한 조건에서 OOM이 발생해 서버가 다운됐습니다. SXSSF(개선)는 평균 응답시간 16.63s로 느리지만 페이징 스트리밍 방식으로 고부하 상황에서도 전 요청이 정상 완료됐습니다.

07

복합 인덱스 적용 — Oracle DBMS_XPLAN 실측 성능 검증

날짜 기반 예약 조회 시 발생하는 Full Scan 문제를 해결하기 위해 복합 인덱스를 적용하고, Oracle DBMS_XPLAN과 SET TIMING으로 실측 검증했습니다.

① RESERVATION 테이블 (10,115건) — CHECKIN_DATE + CHECKOUT_DATE 복합 인덱스 적용 후 Cost 68 → 2 (약 97% 감소)를 확인했습니다.

② ROOM_STOCK 테이블 (9,855건) — (ROOM_TYPE_ID, STAY_DATE) 복합 Unique 인덱스 적용 후 실행시간 0.115초 → 0.042초 (약 63% 단축)를 SET TIMING으로 실측했습니다.
