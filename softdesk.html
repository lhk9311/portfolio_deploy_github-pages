02

Socket.io WebSocket 실시간 통신 — HTTP Polling 대비 선택 근거

GPT 응답은 평균 2~4초 이상의 대기 시간이 발생합니다. HTTP REST 방식은 응답이 완료되어야 클라이언트로 전달되는 Request/Response 구조라, GPT 응답 대기 중 클라이언트 측에서 별도 폴링이 필요하거나 연결이 블로킹됩니다.

Socket.io 기반 WebSocket 연결을 선택하여 서버와 클라이언트 간 상시 연결을 유지함으로써, GPT 응답 대기 중에도 이벤트 기반으로 다른 메시지 송수신이 가능한 구조를 구성했습니다. FAQ 응답(즉시)과 GPT 응답(지연)을 동일한 이벤트 채널로 처리할 수 있어 프론트엔드 구조도 단순화됐습니다.

항목 HTTP REST WebSocket (Socket.io) 채택
연결 방식 요청마다 연결/해제 상시 연결 유지
GPT 대기 중 클라이언트 블로킹 또는 폴링 필요 이벤트 기반으로 다른 처리 가능
관리자 패널 별도 폴링 API 필요 admin-request 이벤트로 실시간 동기화
코드 복잡도 FAQ/GPT 분기마다 별도 endpoint chat message 단일 이벤트로 통합 처리

React 리렌더링 시 이벤트 중복 등록 문제
Socket.io 이벤트 리스너가 컴포넌트 리렌더링 시 중복 등록되어 동일 응답이 여러 번 출력되는 문제가 발생했습니다. useEffect cleanup 함수에서 socket.off("chat response"), socket.off("admin-request")를 명시적으로 호출하여 해결했습니다.

03

GPT 응답 시간 측정 — Date.now() 기반 LLM Latency 분석 & 로딩 UX

LLM API 호출 전후 Date.now()로 응답 시간을 측정하는 로직을 구현해 GPT Fallback 경로의 평균 응답 시간이 2~4초 수준임을 실측했습니다.

이 수치를 바탕으로 GPT 호출 시점에 "🧠 AI 답변 생성중..." 로딩 메시지를 Socket 이벤트로 먼저 클라이언트에 전송하고, 응답 도착 시 로딩 메시지를 실제 답변으로 교체하는 UX 구조를 구성했습니다. 사용자는 응답 대기 중에도 피드백을 받을 수 있어 이탈률을 줄일 수 있습니다.

측정 결과 (로컬 환경 기준)

FAQ 응답
~100ms 이내
GPT Fallback
평균 2~4초

※ Date.now() 기반 실측. gpt-4.1-mini 기준, 네트워크 환경에 따라 변동 가능.

또한 AI 응답에 포함된 \n 줄바꿈 문자가 React에서 렌더링되지 않는 문제를 style={{ whiteSpace: 'pre-line' }}을 메시지 컴포넌트에 적용해 해결하고, GPT 응답 가독성을 개선했습니다.

04

AWS EC2 수동 배포 — Node.js + Nginx + PM2 직접 구성

CIEL 프로젝트에서 Jenkins CI/CD + Docker 자동화 배포를 경험한 후, 이 프로젝트에서는 자동화 없이 서버 구성 전 과정을 직접 수행하며 각 컴포넌트 역할을 체득했습니다.

로컬에서 npm run build로 React를 빌드하고 WinSCP로 EC2에 업로드, Node.js 백엔드는 PM2로 프로세스를 관리해 EC2 재시작 시에도 자동 복구되도록 구성했습니다. Nginx는 포트 8082에서 React 빌드 파일을 서빙하며 백엔드(포트 4000)와 분리된 구조로 운영됩니다.

MySQL 덤프 파일을 로컬에서 생성 후 EC2로 이전하는 과정, .env 파일 EC2 직접 생성을 통한 API 키 보안 관리도 직접 구성했습니다.
